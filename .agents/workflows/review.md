## 사전 확인

`git diff main --stat` 와 `git diff main` 을 실행해서 이번 브랜치의 전체 변경 내용을 파악한다.

## 1단계: Spec 준수 검토

`docs/superpowers/plans/` 디렉토리에서 이번 브랜치에 해당하는 플랜 파일을 찾아 읽는다.

플랜과 구현을 비교해서 다음을 확인한다.

- 플랜의 모든 태스크가 구현됐는가?
- 플랜에 없는 코드가 추가됐는가? (scope creep)
- 각 태스크의 검증 커맨드가 실제로 실행됐는가?

결과를 다음 3단계 심각도로 분류해서 보고한다.

- **Critical**: 머지 불가. 반드시 수정해야 한다.
- **Important**: 출시 전 수정 권장.
- **Nice-to-have**: 선택적 개선 사항.

Critical 또는 Important 항목이 있으면 여기서 멈추고 사용자에게 수정 여부를 확인한다. 모두 해결됐을 때만 2단계로 진행한다.

## 2단계: 코드 품질 검토

diff 전체를 다시 읽고 다음 항목을 확인한다.

- 명백한 버그 또는 로직 오류가 있는가?
- 새로운 로직에 테스트가 빠져 있는가?
- 하나의 파일이 두 가지 이상의 책임을 지고 있지 않은가?
- 기존 프로젝트 컨벤션을 위반하는 코드가 있는가?
- 하드코딩된 시크릿, API 키, 비밀번호가 있는가?

결과를 동일한 3단계 심각도로 분류해서 보고한다.

## 전체 테스트 실행

프로젝트의 전체 테스트 스위트를 실행하고 결과를 확인한다. 실패한 테스트가 있으면 Critical로 분류한다.

## 최종 리뷰 요약 출력

아래 형식으로 최종 결과를 출력한다.

```
## /review 결과

### Spec 준수
- Critical: (없음 또는 항목)
- Important: (없음 또는 항목)
- Nice-to-have: (없음 또는 항목)

### 코드 품질
- Critical: (없음 또는 항목)
- Important: (없음 또는 항목)
- Nice-to-have: (없음 또는 항목)

### 테스트
- 결과: (통과 N개 / 실패 N개)

### 판정
- [ ] Merge 가능
- [ ] 수정 후 재검토 필요
```
