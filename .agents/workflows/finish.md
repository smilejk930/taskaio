## 사전 확인

`git status` 와 `git branch --show-current` 를 실행해서 현재 상태를 확인한다.

커밋되지 않은 변경사항이 있으면 여기서 멈추고 사용자에게 먼저 커밋하거나 stash할지 확인한다.

## 테스트 전체 실행

프로젝트의 전체 테스트 스위트를 실행한다.

실패한 테스트가 있으면 여기서 멈추고 사용자에게 결과를 보고한 뒤 계속 진행할지 확인한다.

## 플랜 완료 여부 확인

`git log main..HEAD --oneline` 을 실행해서 이번 브랜치의 커밋 목록을 확인한다.

`docs/superpowers/plans/` 에서 해당 플랜 파일을 찾아 읽고, 미완료 태스크가 있는지 확인한다. 미완료 항목이 있으면 사용자에게 보고한다.

## 처리 방법 선택

리뷰가 아직 진행되지 않았다면 먼저 `Call /review` 를 실행할지 사용자에게 제안한다.

사용자에게 다음 4가지 옵션 중 하나를 선택하도록 안내한다.

```
1) Merge     — main에 직접 병합
2) PR 생성   — Pull Request를 열 수 있도록 push 후 안내
3) 보류      — 브랜치를 유지한 채 작업 중단
4) 폐기      — 브랜치를 삭제하고 변경사항 버림
```

## 옵션 1: Merge

`git checkout main` → `git merge --no-ff [브랜치명]` 순서로 실행한다.

머지 커밋 메시지는 플랜 파일의 기능 요약을 사용한다.

성공하면 `git branch -d [브랜치명]` 으로 브랜치를 정리한다.

## 옵션 2: PR 생성

`git push origin [브랜치명]` 으로 원격에 push한다.

다음 형식의 PR 본문 초안을 작성해서 사용자에게 제시한다.

```
## 변경 내용
(git log 기반 요약)

## 테스트
- [ ] 전체 테스트 통과
- [ ] 플랜 태스크 완료

## 관련 플랜
docs/superpowers/plans/[플랜파일명].md
```

사용자가 직접 PR을 열 수 있도록 안내한다.

## 옵션 3: 보류

현재 브랜치 이름과 마지막 커밋 SHA를 출력하고, 나중에 재개하는 방법을 안내한다.

## 옵션 4: 폐기

삭제 전에 마지막으로 사용자 확인을 요청한다. 명시적 승인이 있을 때만 진행한다.

`git checkout main` → `git branch -D [브랜치명]` 순서로 실행한다.
