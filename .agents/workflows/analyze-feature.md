# Analyze Feature

## Steps

1. **요구사항 파악**
   사용자의 요청을 자연어 그대로 수용하고, 아래 항목을 질문하여 빠진 정보를 채운다. 한 번에 모든 질문을 하고 사용자가 답하기 전까지 다음 단계로 넘어가지 않는다.
   - 이 기능을 쓰는 사람은 누구인가? (팀원 전체 / 프로젝트 오너만 / 등)
   - 어떤 데이터가 보여야 하는가?
   - 어떤 액션이 가능해야 하는가? (생성 / 수정 / 삭제 / 조회)
   - 엣지 케이스나 제약 조건이 있는가? (예: 삭제 시 연관 데이터 처리)

2. **기존 코드베이스 분석**
   현재 워크스페이스에서 아래 항목을 파악한다. 파악한 내용을 구현에 반드시 반영한다.
   - 연관된 기존 테이블 및 컬럼
   - 재사용 가능한 기존 컴포넌트 (`src/components/`)
   - 재사용 가능한 기존 훅 (`src/hooks/`)
   - 유사한 기존 API Route 또는 Server Action 패턴

3. **DB 변경 필요 여부 판단**
   기존 테이블(`projects`, `members`, `tasks`, `task_dependencies`)로 구현 가능한지 판단한다.
   - 가능: "기존 스키마로 구현 가능합니다" 명시
   - 불가: 추가/변경이 필요한 테이블, 컬럼, RLS 정책을 목록화한다

4. **구현 범위 확정**
   아래 형식으로 구현 범위를 정리한다. 범위가 크면 Phase로 나눠 제안하고 사용자에게 선택하게 한다.

   ```
   📋 구현 범위
   ├── DB 변경: 있음 / 없음
   ├── 새 페이지: /경로 (페이지명)
   ├── 새 컴포넌트: ComponentName.tsx × N개
   ├── 새 API/Action: route 또는 action명
   ├── 재사용: 기존 컴포넌트 또는 훅명
   └── 예상 소요: 크다 / 보통 / 작다
   ```

5. **Feature Spec Artifact 생성**
   아래 구조로 Artifact를 생성한다. 이것이 `/build-feature`에 전달할 설계서다.
   코드 스니펫은 포함하지 않는다. 설계 의도와 구조만 기술한다.

   ```markdown
   # Feature Spec: {기능명}

   ## 목적
   한 문장으로 이 기능이 왜 필요한지 설명

   ## 사용자 시나리오
   1. 사용자가 ... 한다
   2. 시스템이 ... 한다
   3. 사용자가 ... 를 확인한다

   ## DB 변경사항
   - 없음 / 또는 변경 내용 상세

   ## 구현 목록
   ### Pages
   - /경로: 역할 설명
   ### Components
   - ComponentName: 역할 설명
   ### API / Server Actions
   - 엔드포인트 또는 action명: 역할 설명
   ### 재사용
   - 기존 파일명: 어떻게 활용하는지

   ## 엣지 케이스 처리
   - 케이스: 처리 방법

   ## 완료 기준
   - [ ] 확인 가능한 체크리스트 항목
   ```

6. **사용자 승인 요청**
   Artifact 생성 후 반드시 멈추고 아래 메시지를 출력한다. 승인 전까지 코드를 작성하지 않는다.

   ```
   📋 Feature Spec을 작성했습니다.
   검토 후 수정할 부분이 있으면 Artifact에 댓글로 알려주세요.
   승인하시면 이 Spec을 복사해서 새 대화창에서 /build-feature를 실행해주세요.
   ```
