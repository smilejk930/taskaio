## 디자인 문서 확인

`docs/superpowers/` 디렉토리에서 가장 최근 디자인 문서(`*-design.md`)를 찾아 목록을 보여준다.

디자인 문서가 없으면 여기서 멈추고 사용자에게 문서 경로를 직접 알려달라고 요청한다.

디자인 문서가 여러 개이면 어떤 문서를 기준으로 플랜을 작성할지 묻는다.

선택된 디자인 문서를 읽고 전체 내용을 파악한다.

## 파일 구조 맵 작성

플랜 작성 전에 파일 구조 맵을 먼저 결정한다.

생성하거나 수정할 파일 목록과 각 파일의 책임을 한 줄씩 정리해서 사용자에게 제시하고 승인을 받는다.

파일 구조 맵 작성 시 다음 원칙을 따른다.

- 한 파일이 두 가지 이상의 책임을 지면 반드시 분리한다.
- 각 파일은 독립적으로 테스트 가능한 단위로 설계한다.
- 독립적인 서브시스템이 여러 개 있으면 플랜 파일을 분리할 것을 제안한다.

## 태스크 목록 작성

파일 구조가 승인되면 구현 태스크를 작성한다.

태스크 하나는 2~5분 안에 완료 가능한 크기여야 한다. 그보다 크면 분리한다.

각 태스크는 다음 항목을 포함한다.

- **파일**: 생성 또는 수정할 파일 경로
- **구현**: 작성할 코드 또는 변경 내용 (구체적으로)
- **테스트**: 구현 전에 먼저 작성할 실패 테스트
- **검증**: 태스크 완료를 확인하는 커맨드와 기대 출력
- **커밋 메시지**: `[type]: [summary]` 형식

## 플랜 검토 및 저장

플랜 전체를 사용자에게 보여주고 검토를 요청한다.

수정이 필요하면 반영한 뒤 다시 보여준다.

사용자가 승인하면 `docs/superpowers/plans/YYYY-MM-DD-[기능명].md` 경로로 저장하고 git에 커밋한다.
