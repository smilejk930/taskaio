# Feature Spec: DB 커넥션 풀 스케일링 설정 (동시접속 대비)

## 목적
단일 Docker Compose 환경에서 20명 이상의 동시 접속을 원활하게 지원하기 위해, 커넥션 풀의 크기를 고정값이 아닌 환경 변수 기반으로 동적 확장할 수 있도록 구성합니다.

## 사용자 시나리오
1. 시스템 관리자가 `docker-compose.yml` 또는 `.env` 파일을 통해 `DB_POOL_MAX` 값을 20 이상으로 설정한다.
2. 팀원 20명이 동시에 접속하여 간트차트를 조작하거나 업무를 업데이트한다.
3. Node.js는 비동기 모델을 통해 한정된 자원(커넥션 20개 내외)을 재활용하여 대기 없이 요청을 데이터베이스로 전달한다.
4. 로컬 PostgreSQL(기본 최대 커넥션 100) 서버는 20여 개의 커넥션을 안정적으로 유지하며 부하를 버틴다.

## DB 변경사항
- 없음 (순수 풀 설정 개선)

## 구현 목록
### 코어 레이어 (Core Layer)
- **`src/lib/db/index.ts`**
  - 기존 하드코딩된 `max: 8`을 `process.env.DB_POOL_MAX`를 읽어오도록 수정.
  - 올바르지 않은 값 입력 대비 `Number()` 파싱 등 폴백(fallback) 로직 구현. (운영 기본값 20, 개발 기본값 10)

### 배포 구성 (Deployment)
- **`docker-compose.yml`**
  - 앱(taskaio) 서비스의 `environment` 항목에 `DB_POOL_MAX=20` 예시 추가. 

## 엣지 케이스 처리
- **요청 큐잉(Queueing)**: 만약 20개의 커넥션이 모두 바쁜 상태라도 `postgres.js` 드라이버는 자체적으로 큐 방식으로 요청을 대기시킵니다. 따라서 어플리케이션이 죽지 않고 수 ms 딜레이 뒤에 순차 처리됩니다.

## 완료 기준
- [ ] `index.ts` 가 `DB_POOL_MAX` 환경변수를 인식한다.
- [ ] `docker-compose.yml` 에 명시적으로 풀 사이즈 구성을 안내한다.
